# Codex macOS 状态监控与自定义提示音可行性分析

## 1. 目标

在 macOS 上监控 Codex 任务状态，并在以下情况播放提示音：

- Codex 需要用户介入，例如等待授权、回答或其他选择。
- Codex 已经完成任务，产生可以查看的结果。
- 两类提示音可以由用户自行更换。

本文只分析可行性和实现边界，不包含应用制作或代码实现。

## 2. 结论

该需求整体可行。

- “任务完成”可以可靠识别并播放自定义提示音。
- “等待授权”可以可靠识别并播放自定义提示音。
- “等待用户回答普通问题”可以通过 Codex 自带通知可靠提醒，但如果要求第三方程序为它播放独立自定义音效，识别可靠性取决于接入方式。
- 如果只需要声音提醒，优先使用 Codex 官方 Hook，不需要监控数据库或读取对话内容。
- 如果还需要菜单栏人物、任务列表和完整的四状态展示，可以制作只读状态监控器，但它会部分依赖 Codex 当前的本地内部数据格式。

推荐分两阶段实施：

1. 先使用 Hook 完成“介入音”和“完成音”。
2. 确实需要集中状态面板时，再增加菜单栏状态监控器。

## 3. Codex 已有的状态能力

Codex Pet 官方定义了四种任务状态：

| 状态 | 含义 | 外部监控可行性 |
| --- | --- | --- |
| Running | 任务正在执行 | 高 |
| Needs input | 等待用户批准、回答或其他决定 | 授权类高，普通问题类中等 |
| Ready | 任务完成并有未读活动 | 高 |
| Blocked | 任务失败或遇到系统错误 | 中高 |

当多个任务同时有活动时，Codex Pet 的显示优先级是：

1. Needs input
2. Blocked
3. Ready
4. Running

官方参考：[Codex Pets](https://learn.chatgpt.com/docs/pets)

## 4. Codex 自带通知能做到什么

Codex macOS App 已经支持：

- 任务完成通知。
- 授权请求通知。
- 问题通知。
- 控制完成通知从不出现、仅在应用处于后台时出现，或者始终出现。
- 通过 macOS 系统通知权限显示通知和播放系统通知声音。

因此，如果目标只是“不漏掉需要介入和任务完成”，使用 Codex 自带通知就已经能够覆盖主要需求。

当前官方设置说明没有提供分别为“需要介入”和“任务完成”选择本地音频文件的能力。它主要控制通知类别和触发条件，而不是每个事件对应的声音文件。

官方参考：[Codex Notifications](https://learn.chatgpt.com/docs/notifications)

## 5. 方案一：Codex Hook 播放自定义提示音

### 5.1 原理

Codex 官方 Hook 提供与本需求高度匹配的生命周期事件：

- `PermissionRequest`：Codex 请求用户授权时触发。
- `Stop`：一次任务回合停止或完成时触发。

Hook 可以调用一个本地脚本或轻量通知程序，由它根据事件类型播放不同的音频文件：

- `PermissionRequest` 播放“需要介入”提示音。
- `Stop` 播放“任务完成”提示音。

音频文件路径可以保存在用户级配置中，用户更换文件后即可更换声音，不需要修改 Codex 任务本身。

官方参考：[Codex Hooks](https://learn.chatgpt.com/docs/hooks)

### 5.2 优点

- 使用 Codex 官方生命周期接口，稳定性优于解析内部日志。
- 不需要读取提示词、回答内容或工具输出。
- 不需要轮询 SQLite 数据库。
- 资源占用很低。
- 可以分别配置两种提示音。
- 可以全局配置，对不同项目生效。

### 5.3 局限

- `PermissionRequest` 主要覆盖授权场景。
- Codex 主动询问普通问题时，不一定经过 `PermissionRequest`。
- `Stop` 表示回合停止，需要根据 Hook 输入确认它是否属于正常完成；取消或异常停止需要单独判断。
- Hook 命令需要经过 Codex 的信任检查。
- 官方当前说明异步 Hook 尚未支持，因此音频播放程序应尽快返回，避免较长音频阻塞 Hook。

### 5.4 适用结论

如果目标是“声音提醒”，这是首选方案。

建议保留 Codex 自带的“问题通知”，由 Hook 负责自定义授权音和完成音。这样可以避免为了普通问题提醒而依赖内部数据格式。

## 6. 方案二：只读菜单栏状态监控器

### 6.1 本机已确认的数据来源

本机 Codex 当前会保存以下本地状态：

- 线程索引：`~/.codex/state_5.sqlite`
- 会话事件：`~/.codex/sessions/.../*.jsonl`

线程索引中可以取得线程 ID、标题、工作目录、更新时间和对应的会话事件文件路径。

当前会话事件中已确认存在：

- `task_started`
- `task_complete`

因此，可以通过只读方式监听 Codex 会话文件，并推导任务当前状态。

### 6.2 可推导的状态

| 事件或条件 | 推导状态 |
| --- | --- |
| `task_started` | Running |
| 授权请求或等待输入事件 | Needs input |
| `task_complete` | Ready |
| 中止、错误或失败事件 | Blocked |

状态监控器可以常驻菜单栏，显示：

- 当前执行中的任务数量。
- 等待介入的任务数量。
- 已完成但尚未确认的任务数量。
- 被阻塞的任务数量。
- 每个任务的标题、工作目录、更新时间和状态。

当状态发生变化时，可以使用 macOS 原生 `NSSound` 播放用户选择的音频文件。

### 6.3 技术要点

- SQLite 数据库必须只读打开，不能修改 Codex 自己的数据。
- 使用 macOS 文件系统事件监听新增和变化的 JSONL 文件，避免高频轮询。
- 按字节位置增量读取，不重复解析整个会话文件。
- JSONL 最后一行可能尚未写完，需要等换行结束后再解析。
- 多线程同时运行时，需要按线程 ID 分别维护状态。
- 同一个事件可能被多次观察，需要使用事件位置或事件标识去重。
- 音频文件选择结果应持久化；如果启用 App Sandbox，需要保存安全作用域书签。
- 默认只读取事件类型和最少量元数据，不读取或保存对话正文。

### 6.4 主要风险

本地 session JSONL 属于 Codex 内部记录格式。官方 Hook 文档明确提醒 transcript 格式不是稳定接口，未来可能变化。

因此该方案需要：

- 明确绑定当前 Codex 数据结构。
- 遇到未知字段或未知事件时明确报错。
- 不为变量名或字典 key 添加隐式兼容映射。
- Codex 更新后执行一次结构检查，再决定是否调整监控器。

按照当前项目约定，不应同时兼容多个猜测出来的 key。更安全的行为是停止解析并提示“Codex 数据格式已变化”。

### 6.5 适用结论

如果需要独立的菜单栏人物状态、任务列表或全局状态面板，这个方案可行，但维护成本高于 Hook 方案。

## 7. 方案三：Codex App Server

Codex App Server 是官方提供的深度集成接口，可以提供：

- Thread、Turn 和 Item 生命周期。
- `turn/started`、`item/started`、`item/completed` 等流式事件。
- `turn/completed` 最终状态。
- 授权和其他双向交互。

官方参考：[Codex App Server](https://learn.chatgpt.com/docs/app-server)

这是接口层面最完整的方案，但它更适合由自己的客户端启动和管理 Codex 任务。

目前公开文档没有明确保证第三方程序可以直接订阅已经在 Codex macOS App 内运行的全部既有任务。因此：

- 如果未来所有任务都由自己的监控应用发起，App Server 是最佳长期架构。
- 如果只是被动观察现有 Codex macOS App，App Server 不是当前首选。

## 8. 不推荐的方案

### 8.1 通过辅助功能读取 Codex 界面

可以使用 macOS Accessibility API 观察窗口文字或按钮状态，但不推荐：

- 需要较高系统权限。
- UI 文案和层级变化会导致识别失效。
- 多任务和后台窗口识别困难。
- 无法稳定区分完成、等待输入和阻塞。

### 8.2 定时截屏和图像识别

技术上可以识别 Pet 动画或界面状态，但资源消耗、权限成本和误判率都更高，没有采用价值。

### 8.3 修改或替换 Codex 自带声音资源

这可能破坏签名、被应用更新覆盖，也不利于维护，不应采用。

## 9. 提示音自定义设计建议

建议分别配置：

- 需要介入提示音。
- 任务完成提示音。
- 是否启用对应声音。
- 音量。
- 播放测试按钮。
- 是否只在 Codex 位于后台时播放。
- 重复提醒间隔，例如 Needs input 持续未处理时是否再次提醒。

音频格式优先支持 macOS 原生能够直接播放的格式，例如：

- AIFF
- WAV
- MP3
- M4A

不应复制或修改用户选择的源文件；应用只保存文件访问权限和路径信息。

## 10. 推荐实施路线

### 阶段一：最小声音提醒

- 使用用户级 `PermissionRequest` Hook。
- 使用用户级 `Stop` Hook。
- 两个事件分别播放用户选择的声音。
- 保留 Codex 自带问题通知。
- 不读取 SQLite 和 session 正文。

这是工作量最小、隐私风险最低、维护成本最低的方案。

### 阶段二：菜单栏状态

仅在确实需要人物状态面板时增加：

- 菜单栏状态图标。
- 四状态任务列表。
- 只读 SQLite 线程发现。
- JSONL 增量事件监听。
- 状态优先级和重复通知控制。

### 阶段三：完整 Codex 客户端

只有在需要由自己的应用启动、管理和介入 Codex 任务时，才考虑改用 App Server。

## 11. 最终判断

| 需求 | 可行性 | 推荐方式 |
| --- | --- | --- |
| 任务完成播放自定义声音 | 高 | `Stop` Hook |
| 请求授权播放自定义声音 | 高 | `PermissionRequest` Hook |
| 普通问题播放独立自定义声音 | 中 | Codex 自带问题通知，必要时再验证内部事件 |
| 显示 Running / Needs input / Ready / Blocked | 高 | Codex Pet 或菜单栏只读监控器 |
| 监控现有 Codex App 中的所有任务 | 中高 | 本地状态与 JSONL 只读监听 |
| 使用官方稳定 API 管理任务 | 高 | Codex App Server，但任务应由自己的客户端发起 |

综合建议：先采用 Hook 解决自定义声音，只有在确实需要独立人物状态面板时，再制作只读菜单栏监控器。
